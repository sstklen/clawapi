## 品質控制機制（v2 強化版）

每個任務完成後，老大（Opus）自動做：

1. **TypeScript 編譯** — `bun run typecheck` 零錯誤
2. **測試通過** — `bun test` 覆蓋率 >80%
3. **SPEC 對照** — 檢查產出的函數簽名、型別、常數是否跟規格書一致
4. **跨規格一致性** — 確認 @clawapi/protocol 的型別兩邊用法一致
5. **測試案例覆蓋** — 每個任務的「測試案例」欄位全部實作
6. **交接文件** — 每個 Session 結束寫 handoff.md

**v2 新增的品質門檻：**
- 每個任務必須有 `*.test.ts`，否則不算完成
- 共用 test helper（`tests/helpers/`）優先使用，不重複造輪子
- 錯誤碼必須從 `@clawapi/protocol/errors` import，不能硬編碼字串

## v1 → v2 學到的經驗（下次要記住的）

### 🎓 經驗 1：按功能切，不要按部署位置切

規格書按「在哪台電腦跑」分成 A/B/C，但施工任務應該按「做什麼功能」分。
否則橫切關注點（Auth、Config、i18n、Scheduler）一定會掉到縫隙裡。

### 🎓 經驗 2：Phase 0 要夠胖

地基不是只有型別定義。所有 package.json、index.ts 空殼、test helper 都要在 Phase 0 搞定。
否則後面的 Sonnet 連 `bun install` 都跑不起來。

### 🎓 經驗 3：一個任務 = 一個 Sonnet 能獨立完成的單元

如果一個任務需要讀超過 2000 行規格書，或產出超過 2500 行程式碼，就該拆。
Sonnet 的上下文窗口有限，塞太多品質會下降。

### 🎓 經驗 4：預估要 ×1.3

初版預估幾乎必然低估 30%。原因：
- 測試程式碼跟業務程式碼差不多長
- 配置檔案（YAML、JSON）吃行數
- 邊界條件處理比想像中多

### 🎓 經驗 5：驗收標準要可自動化

「通過」太模糊。要寫到「什麼輸入→什麼輸出→用什麼斷言」的程度。
每個任務附帶 3-5 個具體測試案例，Sonnet 才知道「做對了」長什麼樣。
