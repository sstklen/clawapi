## 三、成本預測系統

### 現況
- Token 用量追蹤 ✅（usage_log 有 tokens_input/tokens_output）
- 高用量服務推薦 ✅（engine.ts 建議 4）
- Adapter YAML 有 pricing 欄位 ✅
- **缺的：**
  - 沒有「美元成本估算」（知道用了多少 token，但不知道花了多少錢）
  - 沒有「月度預算預測」
  - 沒有「額度快用完」的警告
  - 沒有「成本比較」（你用 OpenAI 花了 $X，如果用 DeepSeek 只要 $Y）

### 要做什麼

#### 模組 1：成本計算引擎（cost-engine.ts）
```typescript
// 核心資料結構
interface ServicePricing {
  service_id: string;
  model_pattern: string;      // 'gpt-4o*' 匹配模式
  input_per_1m_tokens: number; // $2.50
  output_per_1m_tokens: number; // $10.00
  free_tier_tokens?: number;   // 免費額度
  free_tier_requests?: number; // 免費請求數
}

// 主要方法
class CostEngine {
  // 單次請求成本
  estimateRequestCost(service, model, inputTokens, outputTokens): CostEstimate

  // 近 7 天/30 天總成本
  getWeeklyCost(): CostSummary
  getMonthlyCost(): CostSummary

  // 預測下個月成本（基於趨勢）
  predictNextMonth(): CostPrediction

  // 如果改用別的服務能省多少
  getSavingsComparison(): SavingsReport

  // 額度剩餘估算
  getQuotaRemaining(serviceId): QuotaEstimate
}
```

#### 模組 2：Pricing 資料來源
- 從 Adapter YAML 的 `pricing` 欄位讀取
- 定期從 GitHub 更新（跟 Adapter 市集同步）
- 免費服務標記 `free_tier: true`

#### 模組 3：告警整合
- 月度成本超過閾值 → 通知
- 免費額度快用完 → 通知
- 整合到 growth_guide(view=pool) 顯示

#### 模組 4：MCP 呈現
- growth_guide(view=cost) — 新增一個 cost 視圖
- 顯示：本週花費、月度趨勢、省錢建議、額度剩餘

### 難度：🟡 中

### 怎麼發包

## 四、Webhook 通知系統

### 現況

  - 龍蝦端（個人引擎）的通知設定介面
  - 事件訂閱管理（選擇要收什麼通知）

### 要做什麼

#### 個人引擎端（SPEC-A 範圍）

1. **notification-manager.ts** — 通知管理器
```typescript
class NotificationManager {
  // 通知事件定義
  events = [
    'key.dead',           // Key 死亡（401/403）
    'key.rate_limited',   // Key 被限速
    'quota.low',          // 額度低於 20%
    'quota.exhausted',    // 額度用完
    'service.degraded',   // 服務降級（集體智慧偵測）
    'growth.milestone',   // 成長里程碑（L2 解鎖等）
  ];

  // 發送通知（多管道）
  async notify(event, payload): void

  // 管道：Webhook / MCP 內聯 / CLI 輸出
  channels = ['webhook', 'inline', 'cli'];
}
```

2. **webhook-sender.ts** — HTTP Webhook 發送器
```typescript
class WebhookSender {
  // 發送帶簽名的 Webhook
  async send(url, event, payload): Promise<boolean>

  // HMAC-SHA256 簽名
  sign(payload, secret): string

  // 重試邏輯（3 次，指數退避）
  retry(fn, maxRetries): Promise<boolean>
}
```

3. **設定存儲**
```sql
-- settings 表新增
webhook_url TEXT,          -- 接收通知的 URL
webhook_secret TEXT,       -- HMAC 簽名密鑰
webhook_events TEXT,       -- 訂閱的事件（JSON array）
```

4. **API 端點**
```
GET  /api/settings → 包含 webhook 設定
PUT  /api/settings → 更新 webhook 設定
POST /api/webhook/test → 測試 webhook 連通性
```

### 工時：4 小時
### 難度：🟢 低（結構簡單，就是 HTTP POST + 簽名）

### 怎麼發包
```

  → 寫 notification-manager.ts + webhook-sender.ts + 測試
  → 規格：事件列表 + HMAC 簽名 + 重試邏輯

老大：
  → 整合到 key-pool.ts（Key 狀態變化時觸發通知）
  → 整合到 engine.ts（額度低時觸發通知）
```

## 五、API 文件（Swagger / OpenAPI）

### 現況
- 完全沒有 OpenAPI 定義檔 ❌
- 但 API 已經全部實作完成 ✅
- 路由註釋很清楚 ✅

### 要做什麼

#### 方案選擇

| 方案 | 做法 | 優點 | 缺點 |
|------|------|------|------|
| A. 手寫 OpenAPI YAML | 寫一份 openapi.yaml | 最精確 | 維護負擔大 |
| B. 從代碼生成 | 用 hono 的 OpenAPI 插件 | 自動同步 | 需要改路由定義 |
| C. Scalar + 手寫 | 用 Scalar 做漂亮 UI + 手寫 spec | 好看又精確 | 需要兩邊維護 |

**推薦方案 B**：Hono 有 `@hono/zod-openapi` 插件，可以把路由定義改成帶 schema 的版本，自動產生 OpenAPI 3.0 spec + Swagger UI。

#### 實作步驟

1. 安裝 `@hono/zod-openapi` + `@hono/swagger-ui`
2. 把現有路由改成 OpenAPI 路由格式（加 request/response schema）
3. 掛載 `/docs` 路徑顯示 Swagger UI
4. 掛載 `/openapi.json` 提供 spec 下載

#### 影響範圍
- 改 `api/management.ts`（~30 個路由要加 schema）
- 改 `api/proxy.ts`（OpenAI 相容端點）
- 加 `api/docs.ts`（Swagger UI 掛載）

### 工時：8 小時（大量但機械性的改動）
### 難度：🟡 中（量大但不複雜）

### 怎麼發包
```
Codex --full-auto（最適合，大量重複改動）：
  → 把 30+ 個路由從 app.get/post 改成 OpenAPI 格式
  → 每個路由加 request/response schema
  → 測試：確認 /openapi.json 產出正確

老大：
  → 選定方案（B）+ 安裝依賴
  → 審查 Codex 產出的 schema 是否正確
  → 掛載 Swagger UI
```

### 開工指令
```

```

### 再開工
