## 1. 產品定位

### 一句話描述

**AI API 是一個開源的 AI API 鑰匙管理器 + 智慧路由器。**（產品名稱暫定，最終名稱待決）
龍蝦（AI Agent 使用者）把從各帳號申請到的 API Key 丟進來，系統幫你統一管理、自動切換、智慧調度。

### 為什麼做這個？

- AI Agent 使用者手上有很多 API Key（Groq、Gemini、Brave、DeepL...）
- 免費額度分散在多個帳號，手動管理痛苦
- Key 掛了不知道，呼叫失敗才發現
- 沒有工具幫你把這些 Key 變成一個統一的超級 API

### 類比

> AI API 就像一個**插座轉接器**。
> 你有美規、日規、歐規的充電器（各家 API Key），
> 插進這個萬用轉接頭，
> 出來就是一個統一規格的插座（統一 API）。
> 如果美規的壞了，自動切到日規，你完全不用管。

| Key 池化管理   | 有限             | 有限             | 不行         | 基礎輪換             | **核心功能**         |
| 智慧路由 L3/L4 | 沒有             | 沒有             | 基礎         | 沒有               | **完整四層**         |

| Key 存哪     | 伺服器            | 伺服器            | 他們管        | 環境變數（未加密）        | **本機加密**         |
| 價格         | 企業版收費          | 免費             | 按量收費       | 免費               | **完全免費**         |

```
你的 AI 工具 ──→ AI API（在你電腦上跑）──→ 各家 API 服務
```

> 而且全世界的使用者匿名分享經驗，讓轉接器越來越聰明。

### 3.2 四層服務架構

```
請求進來
  │
  ├─ L1 直轉 ──────────> 指定 API，直接轉發，不加工
  │                      (像郵差，你說寄哪就寄哪)
  │
  ├─ L2 智慧路由 ──────> 自動選最佳 Key + 失敗自動切換
  │                      (像導航，塞車就自動換路)
  │
  ├─ L3 AI 管家 ───────> 龍蝦的金鑰匙(LLM)解讀意圖，
  │                      自動選擇要用哪些工具
  │                      (像管家，你說「幫我查天氣」
  │                       他知道要用哪個 API)
  │
  └─ L4 任務引擎 ──────> 複雜多步驟任務自動執行
                         (像專案經理，拆解→執行→檢查→重試)
```

## 4. 核心功能詳解

### 4.1 Key 池管理

#### 丟 Key 進來的流程

```
龍蝦打開 Web UI (localhost:PORT)
  → 點「新增 Key」
  → 選服務（下拉選單：Groq / Brave / Gemini / ...）
  → 貼上 Key
  → 選池子：🐙 龍蝦王池（我自己的）/ 👨‍👩‍👧 親友分身池（別人給的）
  → 可選：幫 Key 取名字（「媽媽的 Groq」）
  → 送出 → 自動驗證 Key 是否有效 → 完成
```

#### 5 把上限的保護說明（#26 增補）

UI 上的提示文字：

> ⚠️ 每個服務最多 5 把 Key
> 
> 為什麼？保護你。
> 如果你放了 20 把不同帳號的 Key，全部從同一台電腦呼叫同一個 API，
> 供應商可能會發現這個「異常模式」，把你所有的 Key 一起封掉。
> 5 把是安全的上限，讓你的 Key 活得久。

#### Key 健康偵測

- **正常 🟢**：上一次呼叫成功
- **限速 🟡**：收到 429，自動等待冷卻
- **死亡 🔴**：連續 3 次失敗或收到 401/403

偵測是被動的（每次呼叫時偵測），不主動探測。
原因：主動探測浪費免費額度，而且可能被供應商當作異常行為。

### 4.2 智慧路由（L2）

#### Fallback 鏈範例

龍蝦有 Groq、Cerebras、Gemini 的 Key，系統自動排列：

```
LLM 請求進來
  → 試 Groq（速度最快）
    → 成功？回傳 ✅
    → 429 限速？標記限速，往下
  → 試 Cerebras（第二快）
    → 成功？回傳 ✅
    → 死亡？標記死亡，往下
  → 試 Gemini（最後備援）
    → 成功？回傳 ✅
    → 也掛了？回傳友善錯誤訊息
```

路由順序由三個因素決定：

1. **集體智慧建議**（全體龍蝦的統計數據）
2. **龍蝦自己的偏好**（手動調整優先順序）
3. **即時健康狀態**（掛了的自動跳過）

#### 三種路由策略

- **fast**（快）：選延遲最低的
- **smart**（聰明）：綜合考慮成功率、延遲、額度剩餘
- **cheap**（省）：選最便宜/免費的優先

### 4.3 AI 管家（L3）

```
龍蝦問：「幫我查一下東京明天的天氣，然後翻譯成中文」

金鑰匙(LLM) 解讀意圖：
  → 需要：① 天氣查詢 ② 翻譯
  → 用 L2 呼叫天氣 API（Brave Search → 備援 Tavily）
  → 拿到結果
  → 用 L2 呼叫翻譯 API（DeepL → 備援 Google Translate）
  → 整合結果回覆龍蝦
```

**關鍵：金鑰匙是 L3/L4 的大腦。**

- 沒有金鑰匙 → L3/L4 功能關閉，只有 L1/L2
- 金鑰匙推薦用便宜快速的 LLM（Groq Llama、Gemini Flash）
- 金鑰匙的用量清楚顯示：「本次 L3 路由消耗了 150 tokens」

### 4.4 任務引擎（L4）

```
龍蝦：「幫我研究最適合我預算的 GPU」

金鑰匙 規劃步驟：
  Step 1: 搜尋「2026 GPU 性價比排名」（用 Brave/Tavily）
  Step 2: 搜尋「RTX 5070 vs RX 9070 比較」（用 Brave/Tavily）
  Step 3: 搜尋「日本 Amazon GPU 價格」（用 Brave/Tavily）
  Step 4: 整合分析，產出推薦報告（用金鑰匙 LLM）

每一步自動執行 → 失敗自動重試 → 品質不夠自動補充搜尋
```

**L4 成本透明化：**

- 執行前顯示預估：「這個任務大約需要 10-15 次 API 呼叫 + 800 tokens 金鑰匙消耗」
- 執行中即時顯示進度和累計消耗
- 可設定單次任務上限：超過就暫停詢問

### 4.5 Sub-Key 分享

```
龍蝦 A 打開 Web UI
  → 「發行 Sub-Key」
  → 名稱：小明
  → 每日上限：100 次
  → 可用服務：Groq, Brave（或全部）
  → 產生 Sub-Key: sk-oc-a1b2c3d4...

小明拿到 Sub-Key 後：
  → 在自己的工具裡設定 API 地址指向龍蝦 A 的 OpenClaw
  → 用 Sub-Key 當作認證
  → 每次呼叫走龍蝦 A 的 Key 池
  → 龍蝦 A 的 Dashboard 可以看到小明用了多少
```

**安全限制：**

- Sub-Key 是隨機 UUID，不包含任何原始 Key 的資訊
- Sub-Key 無法再轉發（不能發 sub-sub-key）
- 龍蝦隨時可以撤銷任何 Sub-Key
- 撤銷後，進行中的請求會完成，新請求被拒絕

```yaml
    params:
      q: "{{ query }}"
      format: json
    response:
      type: json
      result_path: "RelatedTopics"

  url: "https://api.duckduckgo.com/?q=test&format=json"
```

#### 安全規則

- Adapter YAML **不能執行程式碼**
- Adapter **不能存取其他服務的 Key**（只能引用自己的 service_id）
- 安裝時自動掃描：拒絕任何包含 `{{ key }}` 或 `{{ env.* }}` 的 template
- 認證由引擎注入，Adapter 只宣告 `auth.type`，不碰 Key 值

### 4.8 雲端備份（#32 增補）

```
龍蝦第一次使用：
  → 「要不要用 Google 帳號登入？」
  → 登入後，可選「啟用雲端備份」
  → 設定備份密碼（最少 12 字元，必填）
  → Key 資料用備份密碼 + Google token 雙重加密
  → 加密後的檔案存到 tkman VPS

龍蝦換新電腦：
  → 安裝 OpenClaw
  → 用同一個 Google 帳號登入
  → 輸入備份密碼
  → 自動下載並解密，所有 Key 和設定都回來了

安全重點：
  → tkman 的 VPS 存的是加密後的檔案
  → 沒有備份密碼就無法解密（tkman 也解不開）
  → 忘記備份密碼 = 備份作廢（設計如此，不是 bug）
```

## 5. 資料流與安全模型

### 5.1 Key 加密

```
原始 Key: sk-1234567890abcdef
    │
    ▼ AES-256-GCM 加密
加密 Key: [IV][Auth Tag][Cipher Text]
    │
    ▼ 存入 SQLite
data.db → keys.key_encrypted 欄位
```

加密 Key 來源：裝置特定的隨機產生密鑰，存在 `~/.openclaw/master.key`。
這個密鑰不上傳、不備份（雲端備份用備份密碼另外加密）。

### 5.2 本機 Proxy 認證

```
OpenClaw 啟動時：
  → 檢查 ~/.openclaw/auth.token 是否存在
  → 不存在 → 產生隨機 token，存檔
  → 所有對 localhost:PORT 的請求必須帶此 token

請求格式：
  curl http://localhost:PORT/v1/chat/completions \
    -H "Authorization: Bearer [auth.token的內容]" \
    -H "Content-Type: application/json" \
    -d '{"model": "auto", "messages": [...]}'

為什麼需要？
  → 防止你電腦上的惡意程式偷用你的 Key
  → 瀏覽器擴充套件也可能偷打 localhost
```

### 5.3 API 請求流程（完整）

```
1. 工具 (Claude Code) 發請求到 localhost:PORT
2. OpenClaw 檢查 auth.token ✅
3. 判斷路由層級：
   - 指定 model → L1 直轉
   - model=auto → L2 智慧路由
   - 自然語言問題 → L3 AI 管家（需要金鑰匙）
   - 複雜任務 → L4 任務引擎（需要金鑰匙）
4. 從 Key 池選出最佳 Key
   - 優先：龍蝦王池 > 親友分身池
   - 策略：fast / smart / cheap
5. 用選出的 Key 直接呼叫上游 API
   → 注意：請求從龍蝦電腦直接出去，不經過 tkman VPS
6. 收到回應
   - 成功 → 回傳給工具 + 附帶消耗資訊
   - 失敗 → 自動 failover 到下一個 Key / 下一個服務
7. 非同步記錄用量 + 批次上報匿名統計
```

## 7. 風險與盲點清單

### 7.1 P0 — 必須在架構階段解決

| #   | 風險             | 問題                          | 解法                                                |
| --- | -------------- | --------------------------- | ------------------------------------------------- |
| B1  | Adapter Key 外洩 | 惡意 Adapter 可能讀取其他服務的 Key    | Adapter 只宣告 service_id，引擎注入 Key。靜態掃描拒絕可疑 template |
| B2  | 本機 Proxy 被偷用   | 任何本機程式都可以打 localhost        | 強制 auth.token 認證                                  |
| B3  | 金鑰匙雞生蛋         | 金鑰匙 LLM 限速 → L3/L4 腦死 → 全部掛 | 金鑰匙用量預留 5%，超過自動降級到 L2                             |
| B4  | 雲端備份 Key 託管    | Google token 被盜 → Key 被解密   | 雙重加密：Google token + 使用者備份密碼。缺一不可                  |

### 7.2 P1 — 必須在第一版解決

| #   | 風險                 | 問題                                    | 解法                                                                  |
| --- | ------------------ | ------------------------------------- | ------------------------------------------------------------------- |
| B5  | SQLite 寫入瓶頸        | L4 大量並行請求 → SQLite 鎖死                 | WAL 模式 + 非同步批次寫入                                                    |
| B6  | 集體智慧被灌假資料          | 攻擊者送假統計讓大家避開某服務                       | 信譽加權 + 異常偵測（少數 vs 多數不一致就降權）                                         |
| B7  | Sub-Key 連環轉發       | A 發 key 給 B，B 又發給 C D E               | 架構阻止：Sub-Key 只能在發行者的機器上解析                                           |
| B8  | Debug 困難           | 失敗了不知道是 Key 的問題、API 的問題還是 Adapter 的問題 | 結構化錯誤碼：KEY_EXPIRED / UPSTREAM_429 / ADAPTER_ERROR / NETWORK_TIMEOUT |
| B9  | L3/L4 隱藏成本         | 龍蝦以為一次呼叫，實際金鑰匙被用了很多次                  | 每次回應附帶金鑰匙消耗量 + 可設上限                                                 |
| B10 | Key 輪換時 Sub-Key 斷線 | 龍蝦換了 Key，朋友的 Sub-Key 還指向舊 Key         | `openclaw keys rotate` 自動更新所有映射                                     |

### 7.3 P2 — 第二版前解決

| #   | 風險                   | 問題                         | 解法                                                  |
| --- | -------------------- | -------------------------- | --------------------------------------------------- |
| B11 | 匿名資料側信道洩漏            | 只有一個人用的冷門 Adapter → 可以反推是誰 | 少於 10 人的服務合併到「其他」                                   |
| B12 | Adapter Schema 升級    | v1 → v2 會破壞所有社群 Adapter    | Schema 版本化 + 向後相容 + 自動遷移工具                          |
| B13 | 串流格式差異               | 各家 LLM 的 SSE 格式不同          | 定義標準回應類型：openai_sse / anthropic_sse / google_stream |
| B14 | 離線模式                 | VPS 斷線 → 集體智慧無法同步          | 本機快取路由建議，離線也能用（用舊資料）                                |
| B15 | Failover 換了模型，回答風格變了 | 龍蝦問 Claude，被切到 GPT，回答怪怪的   | 顯示「回應來自 GPT-4o（Claude 目前限速中）」                       |

### 7.4 P3 — 規模大了再處理

| #   | 風險              | 問題                     | 解法                            |
| --- | --------------- | ---------------------- | ----------------------------- |
| B16 | VPS 承受不住 10K 龍蝦 | 每 15 分鐘 × 10K = 大量統計湧入 | 客戶端預聚合 + 二進位協議 + 寫入優化         |
| B17 | 路由建議地區偏差        | 80% 龍蝦在日本 → 建議對巴西沒用    | 按地區分段推薦                       |
| B18 | Adapter 市場變垃圾場  | 500 個 Adapter，大部分壞的    | 健康度評分 + 90 天無人用自動封存           |
| B19 | 社群管理累死 tkman    | Issue 和 PR 太多          | Bot 自動分診 + 志工 moderator + 模板化 |
| B20 | AGPL 嚇跑企業用戶     | 大公司怕 AGPL              | 未來可考慮雙授權（開源 + 商業）             |

### 10.3 災難恢復

```
VPS 掛了怎麼辦？

對龍蝦的影響：
  → 零影響！Key 在龍蝦自己電腦，API 直連上游
  → 只是集體智慧暫時停止更新（用舊的快取）
  → 雲端備份暫時不能同步（但不影響使用）

恢復步驟：
  1. 開新 VPS
  2. 從 S3 備份恢復集體智慧資料庫
  3. 更新 DNS
  4. 完成

自動備份策略：
  - 每日：集體智慧 DB → S3
  - 保留：7 天日備 + 4 週週備
  - 每季一次恢復測試
```

## 附錄 B：CLI 指令速查

```bash
# 啟動
openclaw start                        # 啟動本機引擎

openclaw keys add                     # 互動式新增 Key

# 集體智慧

openclaw telemetry opt-out            # 退出集體智慧

# 備份
openclaw backup export                # 匯出加密備份檔
openclaw backup import backup.enc     # 匯入備份

# 其他

openclaw logs --last 20               # 最近 20 筆使用紀錄
```
